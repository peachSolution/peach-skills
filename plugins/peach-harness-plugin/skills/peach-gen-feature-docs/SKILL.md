# peach-gen-feature-docs — 기능별 명세 생성

기존 기능의 코드 위치와 내용을 `docs/기능별설명/{카테고리명}/{기능명}/` 폴더와, 순번이 포함된 md 파일 4개로 문서화한다.

**파일 순서와 역할** — AI가 코드만 읽는 것보다 구조화된 문서가 효과적인 이유:
- 개요가 진입점 역할을 하여 전체 파일 탐색 없이 기능을 파악
- 로직이 명세보다 앞에 오므로 "왜 이렇게 구현했나"를 먼저 이해하고 코드 수정 시 실수 방지
- 명세의 비즈니스 결정 사항은 코드에서 추론 불가능한 정보를 담음
- TDD 가이드로 AI가 테스트까지 자율 완결

## 이 스킬의 위치 (워크플로우)

```
기존 기능: /peach-gen-feature-docs → Plan Mode → 구현
PRD 필요 시: /peach-gen-feature-docs → /peach-gen-prd → 신규 세션에서 Plan Mode
신규 기능:   /peach-gen-prd → Plan Mode (기능명세 불필요)
```

## When to use

- 기존 기능 개선/수정 전 코드 위치와 내용을 as-is 문서화할 때 (주 용도)
- 신규 기능 구현 완료 후 재수정 가능성이 높은 핵심 기능을 선택적으로 문서화할 때
- 사용자가 "기능 문서화", "기능별 명세 만들어줘", "/peach-gen-feature-docs" 요청 시

> **신규 기능 추가** 시에는 `/peach-gen-prd`를 먼저 사용. 이 스킬은 기존 코드를 분석할 수 있을 때 유효하다.

## Input

- **카테고리명**: 상위 분류 폴더명 (예: 회원관리, 게시판, 결제)
- **기능명(한글)**: 기능 폴더명 (예: 로그인, 비밀번호-변경)
- **한 줄 설명**(선택): 개요.md 요약에 사용

## Workflow

1. **폴더 생성**: `docs/기능별설명/{카테고리명}/{기능명}/`
   - 카테고리 폴더가 없으면 자동 생성
2. **md 파일 생성** (순번 포함, 아래 순서대로):

| 순번 | 파일명 | 역할 |
|------|--------|------|
| 1 | `{기능명}-1-개요.md` | 진입점. 요약, 흐름, 관련 파일, 하위 문서 링크 |
| 2 | `{기능명}-2-로직.md` | 처리 흐름, 에러 코드, 매칭/파싱 규칙, **결정 이유** |
| 3 | `{기능명}-3-명세.md` | 목적, 관련 파일, 결정 사항, 데이터 매핑, 파싱 규칙 |
| 4 | `{기능명}-4-TDD-가이드.md` | 테스트 파일 목록, 실행 명령, 샘플 데이터 |

## 개요 템플릿 ({기능명}-1-개요.md)

```
# {기능명} — 개요

> **읽기 순서**: 개요(현재) → [로직]({기능명}-2-로직.md) → [명세]({기능명}-3-명세.md) → [TDD 가이드]({기능명}-4-TDD-가이드.md)

## 1. 요약
{한 줄 설명 또는 placeholder}

## 2. 전체 흐름
(입력 → 검증 → 저장 등 단계 나열)

## 3. 관련 파일
| 구분 | 경로 |
|------|------|
| Controller (Koa) | api/src/modules/... |
| Service | api/src/modules/... |
| DAO | api/src/modules/... |
| Type | api/src/modules/.../types.ts |
| Store (Pinia) | front/src/modules/.../store.ts |
| Component (Vue) | front/src/modules/.../*.vue |

## 4. 상세 문서
- [로직]({기능명}-2-로직.md) — 처리 단계, 에러 코드, 결정 이유
- [명세]({기능명}-3-명세.md) — 데이터 매핑, 비즈니스 결정, 파싱 규칙
- [TDD 가이드]({기능명}-4-TDD-가이드.md) — 테스트 파일, 실행 명령, 샘플 데이터
```

## 로직 템플릿 ({기능명}-2-로직.md)

섹션 구성:
1. 처리 단계 — 함수별 순서도/테이블
2. 에러 코드 — 코드·설명 테이블
3. **결정 이유** — 주요 구현 결정의 배경. 코드만 보면 알 수 없는 "왜"를 기록
4. 매칭/파싱 로직 — 패턴, 함수명, 예시
5. 코드 경로·함수명

"결정 이유" 섹션이 중요한 이유: AI가 코드를 수정할 때 기존 결정의 맥락을 이해해야 로직을 망가뜨리지 않는다. 예를 들어 "강제 변환 모드가 존재하는 이유"를 모르면 해당 분기를 제거하거나 변경할 수 있다.

## 명세 템플릿 ({기능명}-3-명세.md)

섹션 구성:
1. 개요
2. 관련 파일
3. 결정 사항 — 비즈니스 레벨 결정(코드에서 추론 불가능한 것들)
4. 데이터 매핑(또는 API 명세)
5. 상태/코드 매핑
6. 파싱 규칙
7. 에러 처리

## TDD-가이드 템플릿 ({기능명}-4-TDD-가이드.md)

섹션 구성:
1. 테스트 파일 목록
2. 실행 명령 (`npx vitest run {경로}`)
3. 샘플 데이터 위치
4. Quick reference 표

각 파일은 위 섹션 헤더만 넣고 내용은 `(작성 예정)` 또는 한 두 문장 placeholder로 둔다.

## Reading existing feature docs (AI 지침)

- 기능 수정/분석 시: `@docs/기능별설명/{카테고리명}/{기능명}/` 로 폴더 지정
- 진입: 순번이 가장 낮은 파일(`{기능명}-1-개요.md`)을 먼저 읽기
- 작업 유형별 로드:
  - 비즈니스 로직/에러/수정 → `{기능명}-2-로직.md` 먼저
  - 데이터/매핑/비즈니스 결정 → `{기능명}-3-명세.md`
  - 테스트/샘플 → `{기능명}-4-TDD-가이드.md`
- 순번이 없는 레거시 파일(`{기능명}-개요.md` 등)도 동일한 규칙으로 처리: 개요 → 로직 → 명세 → TDD 순.

## 규칙

- 경로는 저장소 루트 기준 (`api/src/modules/`, `front/src/modules/` 등).
- 카테고리명/기능명에 공백이 있으면 하이픈으로 대체 (예: `비밀번호 변경` → `비밀번호-변경`).
- 순번은 1부터 시작, 항상 `{기능명}-{순번}-{역할}.md` 형식.
- 카테고리 폴더 예시: `회원관리`, `게시판`, `결제`, `상품`, `주문`, `정산`
